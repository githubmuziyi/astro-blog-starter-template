### 快速阅读

本文是 Aaron Levie 关于未来 AI Agent 发展趋势的深刻洞见。随着技术的发展，AI Agent 已经从简单的聊天机器人进化为拥有独立沙箱环境、能够直接编写代码、调用 API 和管理文件系统的智能体（如 Claude Cowork、OpenClaw 等）。未来，企业内的 Agent 数量可能会达到人类员工的 100 倍甚至 1000 倍。这种范式转变意味着软件的主要用户将从人类变为 Agent。

因此，未来的软件必须是“API 优先”的，商业模式也会向基于消费或容量计费的方向演进。同时，这也将催生出全新的 Agent 专属基础设施和工具生态，涵盖算力（沙盒环境）、数据访问、身份认证及合规治理等多个领域。

### 从辅助工具到全能执行者

在过去的几个月里，Agent 领域发生了巨大的变化。去年年底，编程 Agent 已经能够独立完成耗时较长的任务，不再需要人类在开发过程中进行过多干预。

现在的 Agent 不再是配备基础工具的聊天机器人。它们通常拥有自己沙盒化的计算环境，能够针对遇到的任何问题编写并运行代码，直接与 API 和 CLI 交互，甚至拥有自己的文件系统和长期记忆。这种核心能力的提升，加上模型在工具使用和软件开发上的突破，让我们看到了 Agent 能够处理任何抛给它们的任务的曙光。

这种架构最初由 Claude Code、Devin 等编程 Agent 定义，但如今我们已经跨越了鸿沟，将这种能力扩展到了个人体验和知识工作的所有领域（例如 Claude Cowork、Perplexity Computer、Manus，以及运行在自己持久化环境中的 24/7 代理 OpenClaw）。

### 软件主要用户的范式转移

随着能力的快速提升，Agent 将渗透到几乎所有的工作领域中：
*   审查每一份起草的合同
*   处理绝大多数一线客户支持案例
*   审计每家公司的财务状况
*   梳理所有医学研究以用于药物发现
*   生成几乎所有被编写的代码
*   创建大部分销售和咨询演示文稿
*   代表消费者在网络上进行交易

我们不仅会用 Agent 来执行今天已经在做的工作，还会让它们做更多以前无法负担的事情，比如运行模拟、通过不同的选项对我们的想法进行原型设计，并在极低的启动和关闭成本下开展大量新项目。

当把这些加在一起时，我们可以预见，组织中的几乎每个员工都会有许多 Agent 为其工作。未来企业中 Agent 的数量比人类多 100 倍或 1000 倍并不难想象。**随之而来的结果是：数以万亿计的 Agent 将成为未来所有软件的主要用户。**

### 为 Agent 打造它们想要的软件

Paul Graham 曾提出构建软件的最简原则：“打造人们想要的东西”（Make something people want）。这引领了 21 世纪一些最伟大的软件成功故事。

现在，前方的道路变成了**“打造 Agent 想要的东西”**。虽然目前 Agent 的最大用户群体往往是开发者或高度技术化的用户，但在 Agent 为知识工作者完成各种任务的未来，这种由人类主导的工具偏好将慢慢消失。除非企业已经有了硬性标准，否则 Agent 将成为特定工作流采用哪种工具的决策者。

这意味着一切都必须变成 **API 优先（API-first）**。如果你的功能没有 API，那它对 Agent 来说就不存在；如果它不能通过 CLI 或 MCP 服务器暴露，你就会处于劣势。正如 YCombinator 的 Jared Friedman 所警告的：“即使是最好的开发者工具，现在也大多不允许你通过 API 注册账户。这在如今的时代是一个巨大的失误。”如果 Agent 无法轻松注册并开始使用你的服务，你在它们面前就形同虚设。

这还将对商业模式产生重大影响。有些情况下基于用户席位的模式仍然适用，但更多针对 Agent 的用例无法简单地与现有用户挂钩，因为它们的工作量完全不同（例如 Agent 在几秒内完成相当于人类数小时的工作）。任何想要在 Agent 未来存活的工具，都需要在系统中建立某种形式的基于消费或容量的商业模式，甚至支持 Agent 自己为这些服务付费。

### 面向 Agent 的下一代基础设施与工具生态

“把电脑交给人类是个好主意。但把电脑交给电脑，让它们能够像我们一样在电脑上创造产出，这是一个更好的主意。” —— Aravind Srinivas, Perplexity

既然 Agent 拥有自己使用的“计算机”，并且能够编写执行代码、调用技能和外部服务，这就为一套全新的 Agent 技术栈创造了机会。你可以想象一下人类用户在计算机上做什么，Agent 同样需要一套专为它们设计的类似功能。

*   **算力与基础设施**：未来的服务器农场将不再只用于运行我们的应用程序，而是用于运行我们的 Agent（如 E2B, Daytona, Modal 和 Cloudflare 正在推进的沙盒环境）。
*   **数据与文件系统**：Agent 需要访问企业核心文件，并管理自己的记忆数据（正如 Box 正在构建的）。同时 HRIS, CRM, 数据湖等核心企业系统都需要变成 API 优先。
*   **身份与通信**：Agent 需要拥有身份并与外界沟通（如 Agentmail 提供专有邮箱），甚至需要管理自己的预算（通过 Stripe 或 Coinbase 钱包进行微交易）。
*   **安全与合规治理**：当 Agent 访问敏感信息或执行受监管的工作流（如医药或银行业）时，公司需要治理并保留它们所有的工作记录。长驻 Agent 需要身份认证及严格的权限控制。

总而言之，我们显然正在进入一个软件的新时代，我们需要专门为规模化使用的 Agent 设计和构建工具。在这个万亿级 Agent 工作的新世界中，这将引领一种与软件交互的全新方式。
